# Концепция проекта SmartM

## Суть проекта

SmartM - это веб-приложение для самостоятельного бронирования демонстрационных встреч с командой через рабочие календари сотрудников в Mail.ru Calendar.

Клиент заходит на сайт, видит доступные слоты для демо, выбирает удобные дату и время, оставляет контактные данные и получает приглашение на встречу в свой календарь. Команда при этом продолжает вести рабочие календари в Mail.ru Calendar, а SmartM использует их как источник занятости и записывает новые встречи обратно в календарь.

Приложение работает как легкая прослойка между браузером пользователя и календарями Mail.ru: пользователь взаимодействует с простой веб-панелью, а сервер обращается к Mail.ru Calendar по CalDAV.

## Для кого проект

SmartM ориентирован на небольшие команды, которым нужно быстро назначать демонстрационные встречи без ручного согласования времени между клиентом и менеджерами.

Основные роли:

- клиент - заходит на публичную страницу, выбирает свободный слот и бронирует демо;
- сотрудник - ведет свой рабочий календарь в Mail.ru Calendar, а SmartM учитывает его занятость;
- администратор или координатор - управляет сотрудниками, календарями, правилами доступности и параметрами показа слотов.

## Главная идея

Главная идея в том, чтобы клиент сам выбирал дату и время встречи с представителем команды. Менеджеру по продажам больше не нужно вручную согласовывать время с коллегами и клиентом: SmartM берет занятость из календарей, применяет правила доступности и показывает только подходящие слоты.

Источник истины для событий остается в Mail.ru Calendar. SmartM не должен становиться отдельным календарем, а должен помогать клиенту безопасно выбрать время и создать корректную встречу в существующей календарной системе.

## Основные сценарии

1. Настроить сотрудников.
   Администратор добавляет сотрудников, которые могут проводить демо. Для каждого сотрудника указываются имя, e-mail и пароль внешнего приложения Mail.ru. Данные сохраняются локально в конфигурационном файле.

2. Настроить доступность.
   Администратор задает рабочее время, исключенные даты, длительность демо, зазоры между встречами и другие правила показа слотов.

3. Получить занятость сотрудников.
   Сервер подключается к Mail.ru Calendar через CalDAV, получает календари сотрудников, читает события и рассчитывает занятые интервалы.

4. Показать клиенту доступные слоты.
   Публичная страница показывает ближайшие возможные окна для демо. Клиент не видит внутренние события сотрудников, а видит только готовые варианты времени.

5. Забронировать демо.
   Клиент выбирает слот, вводит имя, контакты и, при необходимости, дополнительные данные для подготовки встречи.

6. Создать событие в календаре.
   SmartM создает событие в календаре выбранного сотрудника или команды, добавляет данные клиента и отправляет приглашение участникам.

7. Управлять встречами из админ-панели.
   Администратор может просматривать события, проверять доступность, создавать, редактировать или удалять встречи при необходимости.

## Правила подбора встреч

SmartM может учитывать:

- рабочий интервал времени, например с 09:00 до 18:00;
- исключенные даты, когда встречи не нужно предлагать;
- длительность встречи;
- зазор между встречами;
- занятость одного или нескольких сотрудников;
- доступность конкретного календаря для записи;
- горизонт показа слотов, например ближайшие 3, 7 или 14 дней;
- часовой пояс клиента и команды.

Клиенту должны показываться только слоты, которые реально можно забронировать. Если слот уже занят в Mail.ru Calendar, он не должен попадать в публичную выдачу.

## Публичная часть

Публичная часть SmartM должна быть максимально простой:

- список доступных дат и времени для демо;
- форма бронирования;
- подтверждение выбранного времени;
- понятное сообщение об успешной записи;
- приглашение в календарь клиента.

Публичная страница не должна показывать внутренние календари, имена всех сотрудников без необходимости, пароли, служебные ошибки CalDAV или технические детали интеграции.

## Админ-панель

Админ-панель нужна для настройки и контроля работы публичного бронирования. В ней должны быть доступны:

- список сотрудников;
- подключение календарей Mail.ru;
- проверка событий и занятости;
- настройка правил доступности;
- просмотр найденных слотов;
- ручное создание, редактирование и удаление встреч.

## Архитектура

Проект осознанно остается простым:

- backend: `index.js` на Node.js и встроенном HTTP-сервере;
- frontend: plain HTML, CSS и browser JavaScript в папке `public/`;
- хранение: локальный JSON-конфиг в `DATA_DIR`;
- интеграция: Mail.ru Calendar через CalDAV;
- внешние runtime-зависимости: отсутствуют.

## Данные

Конфигурация хранит список сотрудников:

- `id`;
- `name`;
- `email`;
- `password`;
- `activeEmployeeId`.

Для клиентского бронирования дополнительно понадобится хранить или передавать:

- имя клиента;
- контактный e-mail клиента;
- выбранный слот;
- тему демо;
- комментарий или дополнительные данные клиента;
- статус создания календарного события.

Пароли являются чувствительными данными. Их нельзя логировать, показывать в ошибках или коммитить в репозиторий вместе с локальными конфигами.

## Роль backend

Backend отвечает за:

- отдачу статических файлов интерфейса;
- чтение и сохранение конфигурации;
- нормализацию старого и нового формата конфига;
- авторизацию в Mail.ru Calendar через Basic Auth;
- discovery CalDAV-адресов;
- получение календарей и событий;
- разбор XML и ICS;
- создание, обновление и удаление событий;
- расчет занятости и свободных слотов;
- выдачу публичных доступных слотов без раскрытия внутренних событий;
- создание встречи по заявке клиента;
- подготовку календарного приглашения для клиента и сотрудника.

## Роль frontend

Frontend отвечает за:

- публичный выбор даты и времени клиентом;
- форму бронирования демо;
- показ подтверждения записи;
- отображение сотрудников;
- переключение активного сотрудника;
- формы добавления и редактирования данных;
- загрузку календарей и событий через API;
- отображение событий, свободных слотов и общих слотов встреч;
- применение пользовательских правил подбора времени.

Со временем frontend логически стоит разделить на две зоны:

- публичную страницу бронирования для клиента;
- закрытую админ-панель для команды.

## Ограничения проекта

SmartM не является полноценной CRM, корпоративной системой бронирования или отдельным календарным сервисом. Он не должен дублировать Mail.ru Calendar как источник истины.

Главное ограничение и одновременно преимущество проекта - простота. Новые функции стоит добавлять так, чтобы не превращать приложение в тяжелую платформу без необходимости.

SmartM также не должен раскрывать клиенту внутреннюю структуру команды и календарей. Публичная часть показывает только итоговую доступность, а не причины занятости.

## Направление развития

Возможные следующие шаги:

- выделить публичную страницу бронирования демо;
- добавить закрытую админ-панель;
- добавить создание приглашения для клиента после выбора слота;
- улучшить безопасность хранения учетных данных;
- добавить авторизацию в сам SmartM;
- расширить правила подбора встреч;
- добавить более удобный выбор диапазона дат;
- улучшить обработку повторяющихся событий;
- добавить уведомления и подтверждения записи;
- добавить защиту от двойного бронирования одного слота;
- подготовить конфигурацию под новый сервер развертывания.
